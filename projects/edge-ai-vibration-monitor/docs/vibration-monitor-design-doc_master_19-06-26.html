On-Device Machinery-Health Monitor

ESP32 + ADXL345 + Edge Impulse. Learn a machine’s vibration signature, classify its health state on-device, and report a verdict — not a waveform. The comms-denied subsea platform as the design constraint.

Phase: Design / Early Build ● v1 Scope Set ⚠ 8 hrs/week
Honest Scope — what v1 actually claims no inflation, any channel

The long-term goal is catching faults while they are still developing. v1 is precise about what it proves: it classifies present, established fault states — imbalance, looseness, and a seeded bearing defect — end-to-end on-device. Genuinely incipient bearing detection lives in high-frequency resonance (envelope-analysis territory) above the v1 sensor’s usable band — a documented v2 target, not a v1 result. Nothing written here may drift ahead of what is built.

Design a repeatable interface, measure how coupling shifts the features, fix a mounting spec. This is the headline thread (below).

Headline Decision — the thread only this background can author ★ Centrepiece

The mount is part of the measurement — so make it a controlled variable, not an afterthought.

A vibration sensor measures the machine through its mount. Stiffness, contact area, preload and location form a mechanical filter that changes the measured spectrum for the same fault — which is exactly why industrial condition-monitoring obsesses over stud-mounting and accelerometer siting. A fabrication background turns this from a nuisance into a controllable, ownable variable: build the fixture, measure how coupling moves the features, define a mounting spec so a model can transfer between machines. The pure-software route stops at “feed the FFT to a classifier”; the edge no software-only candidate replicates is owning the mechanical coupling end-to-end.

Why it’s pulled forward (the correction that mattered). This was sitting as a deferred open question while the commodity FFT+ML work was front-loaded — which inverts the advantage. It is now ADR-0009 and a Phase-1 deliverable: a coupling sensitivity study (vary mount against one fixed fault, commit the spectra) before the fault dataset is built. Cheap, possible now on the dev-board rig, and the single most differentiating result in the repo.

Direct analogue of the buoy. On the buoy, the fabricated hull shapes the heave signal the DSP sees; here the fabricated mount shapes the vibration signal. Same closed loop — build the physical thing, measure how it filters the signal, correct/standardise in software — carried across both projects.

Prior Art & Positioning — disarm the “don’t you know about…” question editable

Framing: “This is established practice — my specific angle is owning the mechanical coupling on a comms-denied, power-limited edge node.”

  • CWRU Bearing Data Center — the standard public bearing-fault dataset (12/48 kHz, high-bandwidth sensor). Used as a pipeline cross-check by down-sampling into the v1 band — not a parity claim against CWRU-grade kit.
  • ISO 10816 / 20816 (vibration severity) — the established framework for machine vibration limits. Reference point for “doing it properly”; cite to show the method sits on standards, not invented thresholds.
  • Commercial CbM nodes (e.g. piezo wireless sensors) — mains/industrial condition-monitoring is mature. Position against on cost, on-device inference, and the comms-denied subsea constraint they don’t target.
  • Envelope / demodulation analysis — the classical technique for early bearing detection. Named as the v2 route precisely because v1 bandwidth can’t reach it — signals awareness, not ignorance.
